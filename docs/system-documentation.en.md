# Perazzo API System Documentation

This document explains how the backend works, how the main business flows are implemented, and which files are responsible for each step. All API routes are mounted under the `/api/v1` prefix by `app/main.py` and `app/api/v1/routes.py`.

## Architecture Overview

The backend is a FastAPI application organized around routers, services, schemas, and SQLAlchemy models.

- `app/main.py` creates the FastAPI app, configures CORS, registers the v1 router, and exposes `/health`.
- `app/api/v1/routes.py` registers every module router.
- `app/api/v1/routers/*.py` defines HTTP endpoints and request/response boundaries.
- `app/services/*.py` contains business rules, persistence orchestration, validations, and serialization.
- `app/domain/models/*.py` defines SQLAlchemy database models.
- `app/schemas/*.py` defines Pydantic payloads and response contracts.
- `app/core/config.py` loads environment variables through `pydantic-settings`.
- `app/core/database.py` creates the SQLAlchemy engine and request-scoped sessions.
- `app/core/dependencies.py` implements authenticated user resolution from bearer tokens.
- `migrations/` contains Alembic migrations.

The backend is store-scoped. Most authenticated entities belong to one `store_id`, and services resolve that store through `StoreService.get_by_user_id(current_user.id)`.

## Authentication and Account Flow

Account creation starts at `POST /api/v1/auth/register`, implemented in `app/api/v1/routers/auth.py`. The router receives `UserCreate`, calls `UserService.create()` in `app/services/user.py`, hashes the password through `app/core/security.py`, stores the user, and generates an email verification token with `create_email_verification_token()` from `app/util/jwt.py`.

Login uses `POST /api/v1/auth/login`. The router calls `UserService.authenticate()`, which loads the user by email and verifies the password. If valid, the router returns a bearer token created by `create_access_token()`. The token payload stores the user id in `sub`.

Protected endpoints depend on `get_current_user()` from `app/core/dependencies.py`. That dependency reads the `Authorization: Bearer <token>` header, decodes the token with `decode_access_token()`, validates the `sub` UUID, and loads the user with `UserService.get_by_id()`.

Email verification is handled by `POST /api/v1/auth/email/verify`. The endpoint decodes the email token, loads the user, marks `is_email_verified=True`, and clears `email_verification_token`.

Password recovery uses `POST /api/v1/auth/password/forgot` to generate a `reset_password_token`, persist it on the user, and send a password reset email. The endpoint is implemented in `app/api/v1/routers/auth.py`; it calls `create_password_reset_token()` from `app/util/jwt.py` and `send_password_reset_email()` from `app/services/email.py`. The API never returns the token to the client. It returns a generic message so callers cannot discover whether an email is registered.

Password reset links are built from `FRONTEND_URL` and point to `/reset-password?token=<token>`. SMTP settings are loaded in `app/core/config.py` through `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM_EMAIL`, `SMTP_FROM_NAME`, and `SMTP_USE_TLS`. In local development, if SMTP is not configured, `app/services/email.py` writes the reset link to the API log instead of returning it in the HTTP response. In production, SMTP should be configured through Render environment variables, for example with Resend.

`POST /api/v1/auth/password/reset` receives the token and the new password, decodes the token with `decode_password_reset_token()`, validates that it matches the user's latest saved `reset_password_token`, validates the password rules through `validate_password_rules()`, replaces the password hash through `hash_password()`, clears `reset_password_token`, and commits the change.

User profile updates are handled by `PUT /api/v1/auth/me`, which requires authentication and delegates to `UserService.update()`.

## Store Flow

The authenticated store endpoints live in `app/api/v1/routers/store.py`.

- `POST /api/v1/store` creates the current user's store.
- `GET /api/v1/store/me` returns the current user's store.
- `PATCH /api/v1/store/me` updates store settings.
- `PATCH /api/v1/store/me/today-open` toggles the current weekday open/closed state.
- `GET /api/v1/store/{slug}` returns a store by slug.

Business rules are in `app/services/store.py`. A user can only have one store. Store slugs are generated by `app/util/slug.py`. Business hours are normalized and validated by `app/util/store_hours.py`. Store serialization also computes `is_open_now`, which is used by the catalog to decide if customers can add to cart or checkout.

## Product and Category Flow

Products are implemented by `app/api/v1/routers/product.py` and `app/services/product.py`.

- `POST /api/v1/products` creates a product.
- `GET /api/v1/products` lists products with pagination, search, category filter, sorting, and `X-Total-Count`.
- `GET /api/v1/products/{slug}` returns a product by slug.
- `PATCH /api/v1/products/{product_id}` updates a product.
- `DELETE /api/v1/products/{product_id}` soft-disables the product and removes it from carts.

Products are scoped to the current user's store. Product slugs are unique per store. If stock is set to `0`, the product is automatically marked inactive. Catalog queries only return active products and, in catalog mode, products with stock available or unlimited stock.

Categories are implemented by `app/api/v1/routers/category.py` and `app/services/category.py`.

- `POST /api/v1/categories` creates a category.
- `GET /api/v1/categories` lists categories ordered by `sort_order` and name.
- `GET /api/v1/categories/{slug}` returns a category.
- `PATCH /api/v1/categories/{category_id}` updates a category.
- `POST /api/v1/categories/reorder` persists manual ordering.
- `DELETE /api/v1/categories/{category_id}` deletes a category.

Categories are store-scoped. New categories receive the next `sort_order`. Product-category relationships are defined in `app/domain/models/product_category.py`.

## Public Catalog Flow

Public catalog endpoints are implemented in `app/api/v1/routers/catalog.py`. They do not require authentication, but they only work for stores where `has_catalog_active` is true.

- `GET /api/v1/catalog/{store_slug}/home` returns store data, categories, and home sections.
- `GET /api/v1/catalog/{store_slug}/products` lists public catalog products.
- `GET /api/v1/catalog/{store_slug}/categories/{category_slug}` lists products for a category.
- `GET /api/v1/catalog/{store_slug}/products/{product_slug}` returns a public product page.
- `GET /api/v1/catalog/{store_slug}/payment-methods` lists payment methods for checkout.
- `GET /api/v1/catalog/{store_slug}/delivery-methods` lists delivery methods for checkout.

The catalog router uses `StoreService`, `CategoryService`, `ProductService`, `PaymentMethodService`, and `DeliveryMethodService`. It serializes a smaller public shape through helpers inside `catalog.py`. Cart mutation and checkout endpoints call `_ensure_store_open_for_cart_actions()`, which blocks cart actions when the store is currently closed.

## Cart Flow

There are two cart surfaces: authenticated dashboard carts and public catalog carts. Both use `CartService` from `app/services/cart.py`.

Authenticated cart endpoints are in `app/api/v1/routers/cart.py`:

- `POST /api/v1/carts` creates a cart with the first product.
- `GET /api/v1/carts` lists carts.
- `GET /api/v1/carts/{cart_id}` returns a cart.
- `PATCH /api/v1/carts/{cart_id}` updates customer data, delivery choice, payment method, or appends products.
- `PUT /api/v1/carts/{cart_id}/products` replaces the product list.
- `POST /api/v1/carts/{cart_id}/checkout` turns the cart into an order.
- `DELETE /api/v1/carts/{cart_id}` deletes the cart.

Public catalog cart endpoints are in `app/api/v1/routers/catalog.py`:

- `POST /api/v1/catalog/{store_slug}/carts`
- `GET /api/v1/catalog/{store_slug}/carts/{cart_id}`
- `PUT /api/v1/catalog/{store_slug}/carts/{cart_id}/products`
- `POST /api/v1/catalog/{store_slug}/carts/{cart_id}/preview-total`
- `POST /api/v1/catalog/{store_slug}/carts/{cart_id}/checkout`
- `DELETE /api/v1/catalog/{store_slug}/carts/{cart_id}`

`CartService.create()` validates that the product exists, belongs to the store, is active, and has enough stock. `replace_products()` deletes the cart when the product list becomes empty. `checkout()` requires products, customer name, customer phone, and payment method. Public catalog checkout additionally validates delivery method and address when delivery is selected, resolves the payment method by id, creates an `OrderCreate` payload, delegates to `OrderService.create()`, and deletes the cart.

## Order Flow

Orders are implemented by `app/api/v1/routers/order.py` and `app/services/order.py`.

- `POST /api/v1/orders` creates an order.
- `GET /api/v1/orders` lists orders for a date, with pagination and optional search.
- `GET /api/v1/orders/search` searches by order number, customer name, customer phone, or product name.
- `GET /api/v1/orders/{order_id}` returns one order.
- `PATCH /api/v1/orders/{order_id}` updates order products, customer data, delivery, courier, and payment data.
- `DELETE /api/v1/orders/{order_id}` deletes an order.
- `PUT /api/v1/orders/{order_id}/status` changes status.
- `POST /api/v1/orders/preview-total` calculates totals before saving.

`OrderService.create()` resolves the store, loads active products, generates a short order number, gets or creates a customer by phone, resolves delivery method, optionally assigns a courier, calculates item totals and delivery total, and stores the order with status `pending`.

Order status changes drive stock and customer metrics. In `OrderService.update_status()`, moving to `confirmed` or `deliveried` reduces stock once. Moving to `canceled` restores stock if it had been reduced. Moving into `deliveried` increments customer delivered order counters and total spent; moving out of `deliveried` decrements those metrics.

Order listing defaults to the current date in the `America/Sao_Paulo` timezone. Serialization includes product details, customer metrics, delivery method, courier, status, payment method, observation, total, and timestamps.

## Customers

Customer endpoints are implemented by `app/api/v1/routers/customer.py` and `app/services/customer.py`.

- `POST /api/v1/customers`
- `GET /api/v1/customers`
- `GET /api/v1/customers/{customer_id}`
- `PATCH /api/v1/customers/{customer_id}`
- `DELETE /api/v1/customers/{customer_id}`

Customers are store-scoped. Listing returns `orders_count` through a subquery. Deletion is careful: orders are reassigned to a fallback deleted customer instead of losing historical order links.

## Payment and Delivery Methods

Payment methods are implemented by `app/api/v1/routers/payment_method.py` and `app/services/payment_method.py`.

- `POST /api/v1/payment-methods`
- `GET /api/v1/payment-methods`
- `GET /api/v1/payment-methods/{payment_method_id}`
- `PATCH /api/v1/payment-methods/{payment_method_id}`
- `DELETE /api/v1/payment-methods/{payment_method_id}`

Payment method names must be unique per store.

Delivery methods are implemented by `app/api/v1/routers/delivery_method.py` and `app/services/delivery_method.py`.

- `POST /api/v1/delivery-methods`
- `GET /api/v1/delivery-methods`
- `GET /api/v1/delivery-methods/{delivery_method_id}`
- `PATCH /api/v1/delivery-methods/{delivery_method_id}`
- `DELETE /api/v1/delivery-methods/{delivery_method_id}`

Delivery methods represent neighborhoods or delivery areas with a price. They are required for delivery orders and are included in order total calculations.

## Cash Register

Cash register endpoints are implemented by `app/api/v1/routers/cash_register.py` and `app/services/cash_register.py`.

- `GET /api/v1/cash-register/summary`
- `POST /api/v1/cash-register/entries`
- `PATCH /api/v1/cash-register/entries/{entry_id}`
- `DELETE /api/v1/cash-register/entries/{entry_id}`

The summary combines automatic order revenue with manual entries, manual expenses, and profit withdrawals. `CashRegisterService.get_summary()` supports day, week, month, and year views. It groups values by payment method and calculates totals such as auto entries, manual entries, expenses, profit withdrawals, delivery total, and balance.

## Couriers

Courier endpoints are implemented by `app/api/v1/routers/courier.py` and `app/services/courier.py`.

- `POST /api/v1/couriers`
- `GET /api/v1/couriers`
- `PATCH /api/v1/couriers/{courier_id}`
- `DELETE /api/v1/couriers/{courier_id}`
- `POST /api/v1/couriers/adjustments`
- `PATCH /api/v1/couriers/adjustments/{adjustment_id}`
- `DELETE /api/v1/couriers/adjustments/{adjustment_id}`
- `GET /api/v1/couriers/summary`

Couriers can be linked to delivery orders. The summary calculates delivery count, delivery amount, manual adjustments, unassigned deliveries, and total courier earnings for day, week, month, or year periods.

## Data Ownership Rules

The core safety rule is store isolation. Services resolve `store_id` from the authenticated user or receive an explicit `store_id` for public catalog flows. Product, category, customer, order, cart, payment method, delivery method, courier, and cash register queries always filter by store.

Authenticated routers use `dependencies=[Depends(get_current_user)]`, while public catalog routes validate the store by slug and catalog activation.
